Coding standards

Naming conventions, file structure & DRY principles — SwiftUI edition

Naming rules
Views — noun phrase, no "View" suffix
Use TransactionList, not TransactionListView — redundant with the struct context
ViewModels — "ViewModel" suffix, never "Manager"
LoginViewModel is clear; LoginManager is ambiguous
Booleans — is / has / should prefix
isLoading, hasError, shouldRefresh
Extensions — group by protocol or concern in separate files
Date+Formatting.swift, View+ErrorAlert.swift
Do / Don't
✓ do
@Published var isLoading = false

func fetchTransactions() async {
  isLoading = true
  defer { isLoading = false }
  // ...
}
✗ don't
// primitive flag, no async
var loading = false

func getData() {
  loading = true
  // no defer
}
File structure
Recommended layout Swift
// App/ // ├── Core/ → utilities, extensions // ├── Features/ → one folder per feature // │ └── Login/ // │ ├── LoginView.swift // │ ├── LoginViewModel.swift // │ └── LoginModel.swift // ├── Services/ → API, persistence // └── Resources/ → assets, localisation
Key patterns
@MainActor on all ViewModels
Guarantees @Published updates happen on the main thread — no manual DispatchQueue.main.async
Protocol-driven services
Define protocol TransactionServiceProtocol so ViewModels can be unit-tested with mock services
@ViewBuilder for sub-components
Extract repeated UI into @ViewBuilder private func rowView(...) before promoting to a new file
Naming rules
Composables — PascalCase noun phrase, no "Composable" suffix
Use TransactionRow, not TransactionRowComposable
ViewModels — "ViewModel" suffix
TransactionViewModel owns the UiState; screens call collectAsStateWithLifecycle()
UiState — data class with "UiState" suffix
TransactionUiState with isLoading, transactions, errorMessage
Screen vs Component separation
TransactionScreen takes ViewModel; TransactionContent is stateless and previewable
Do / Don't
✓ do
data class TransactionUiState(
  val isLoading: Boolean = false,
  val error: String? = null
)
val ui by vm.uiState
  .collectAsStateWithLifecycle()
✗ don't
// mutable var, not StateFlow
var isLoading = false

// LiveData instead of StateFlow
val data: LiveData<List<Tx>>
File structure
Recommended layout Kotlin
// app/src/main/java/com.example/ // ├── core/ → utilities, extensions // ├── data/ → repository, data sources // ├── domain/ → use cases, models // ├── ui/screens/ // │ └── transaction/ // │ ├── TransactionScreen.kt // │ ├── TransactionViewModel.kt // │ └── TransactionUiState.kt // └── di/ → Hilt modules
Key patterns
collectAsStateWithLifecycle() — not collectAsState()
Stops collecting when the composable leaves composition, preventing background leaks
Hoisted state — screens stateful, content stateless
TransactionScreen collects from ViewModel; TransactionContent(uiState, onRetry) is pure
rememberSaveable for ephemeral UI state
Search query, scroll position — rememberSaveable survives process death

UiState + ViewModel
TransactionViewModel.kt
Kotlin
data class TransactionUiState( val transactions: List<Transaction> = emptyList(), val isLoading: Boolean = false, val errorMessage: String? = null ) @HiltViewModel class TransactionViewModel @Inject constructor( private val repository: TransactionRepository ) : ViewModel() { private val _uiState = MutableStateFlow(TransactionUiState()) val uiState: StateFlow<TransactionUiState> = _uiState.asStateFlow() fun fetchTransactions() { viewModelScope.launch { _uiState.update { it.copy(isLoading = true) } repository.getTransactions() .onSuccess { txs -> _uiState.update { it.copy(transactions = txs, isLoading = false) } } .onFailure { e -> _uiState.update { it.copy(errorMessage = e.message, isLoading = false) } } } } }

Code review checklist

What to look for before approving a PR

Review layers
1 · Correctness first
Does it compile? Are edge cases (empty, error, loading) handled? No force-unwraps in production paths.
2 · Architecture compliance
Business logic in ViewModel, not in View. No direct API calls from a View body.
3 · Naming & readability
Would a junior reading this file tomorrow understand every identifier without the PR description?
4 · Performance signals
No unnecessary redraws. Heavy work off the main thread. Pagination for lists > 50 items.
5 · Test coverage
At minimum, ViewModel unit tests for happy path + error path. UI tests for critical flows.
Quick checklist
No @State for server data — use @Published in ViewModel
Empty, loading & error states all handled
No business logic in View body
Dark mode & different text sizes tested
async/await used for all network calls
Destructive actions have confirmation dialog
ViewModel unit tested
No hardcoded strings — use localisation
Red flags — instant reject
Force-unwrap (!) on optional that could be nil in production
Use guard let, if let, or ?? with a meaningful default
API call directly inside a View body or init()
Move to .task modifier or ViewModel's fetchOnLoad()
Hardcoded secrets or API keys in source files
Use env variables, .xcconfig / BuildConfig, or a secrets manager
